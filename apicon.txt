environment.*
authLoginPath: '/auth/login'

Routes: /login (guest only), /employees/* (requires login). Log out appears in the header when signed in.

Default login URL: POST /api/auth/login with body { "username": "...", "password": "..." }.
If your API differs, change authLoginPath in environment.development.ts (e.g. /Auth/login) and/or the LoginRequest fields in auth.model.ts.

Token in response: first non-empty of token, accessToken, or access_token. Add another property in extractAccessToken if your API uses a different name.

Other PC API: set apiBaseUrl in environment.development.ts to that host (e.g. http://192.168.x.x:7065/api) and ensure CORS allows your Angular origin, or point proxy.conf.json target at that machine when using ng serve.


Login page (/login)
Reactive form posts to AuthService.login(), which calls POST {apiBaseUrl}{authLoginPath} (default /api/auth/login).
On success, the JWT is stored in localStorage under auth_access_token, and isAuthenticated is set to true.
Errors use getApiErrorMessage() so your backend exception text is shown.

Route guards

authGuard: employee routes check for a stored token; if missing → redirect to /login.
guestGuard: if you already have a token → redirect to /employees (so you do not see login again while logged in).
HTTP interceptor (authInterceptor)
For requests to your API (URLs under /api or apiBaseUrl), it adds:
Authorization: Bearer <token>
Login is excluded so the first request does not send an old token.
On 401 from a protected call, it clears the token and sends you to /login.

Logout
Header Log out calls AuthService.logout() → removes token, sets isAuthenticated to false, navigates to /login.

Persistence
Refreshing the page keeps the token in localStorage, so authGuard still allows access until you log out or the API returns 401 (expired/invalid token).

You do not duplicate JWT logic on the frontend beyond storing the string and attaching it; validation stays on the backend.
